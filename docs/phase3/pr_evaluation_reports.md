# Pull Request Evaluation Reports

## Purpose

Pull requests receive a concise ExperimentOS AI quality report from CI when GitHub permits a
comment. The comment is informational; the existing `ai-quality-gate` check remains the only
quality check that controls merge eligibility.

## Architecture

The reporting layer consumes JSON generated by ExperimentOS:

- `phase3/quality_policy.json` is the authoritative policy outcome.
- `phase3/ai_quality_gate.json`, `artifact_manifest.json`, and `ci_environment.json`
  provide execution and artifact metadata.
- Existing suite JSON reports provide compact suite details and compatible metric values.

It writes a normalized `pr_quality_report.json`, job-summary Markdown, and
`pr_comment.md`. It does not parse human Markdown, rerun evaluations, recompute thresholds, or
change baselines.

## Reviewer output

The job summary and marker-owned PR comment contain:

- overall policy status and version
- retrieval, answer quality, workflow, prompt, factuality, and reliability category status
- critical findings, warnings, skipped judge metrics, compatible metric changes, and suite status
- offline execution declarations, artifact name, run identifier, and a pointer to full artifacts

The marker is `<!-- experimentos-ai-quality-report -->`. The comment publisher lists comments,
updates the existing bot-authored marked comment, or creates one when none exists. Unrelated
comments are never changed.

Comments are deliberately bounded by finding, warning, suite, and character limits. When content
is omitted the report states how many entries were omitted and directs reviewers to artifacts.

## Metric deltas

A delta is shown only when an explicitly provided baseline has the same quality-policy and dataset
version as the current structured report. Each displayed row contains current value, baseline,
absolute delta, policy threshold, and status. Missing or incompatible baselines are labelled
unavailable. A score delta is not a claim of statistical or causal improvement.

Baselines are never updated automatically.

## Permissions and fork safety

The evaluation jobs remain read-only. A separate `pr-quality-comment` job runs only for
`pull_request` events and requests only `contents: read`, `actions: read`, and
`pull-requests: write`. Push and dispatch events do not run this job and never receive comment
write permission.

Forked pull requests still run evaluation, job-summary generation, and artifact upload. GitHub
normally supplies a read-only token for an untrusted fork, so comment publication can be
unavailable; that result is informational and cannot hide the report or alter the gate verdict.

The workflow does not use `pull_request_target`. It never runs untrusted pull-request code with
a write token or repository secrets.

## Failure propagation

The quality-gate command records its exit code, then report generation, summary publication, and
artifact upload run with `always()`. The last gate step exits with the captured code. Therefore
policy failures remain non-zero, infrastructure failures remain non-zero, and successful reporting
cannot turn either into a pass.

The comment job is intentionally not a required branch-protection check. GitHub API outages and
fork token restrictions must not block a valid merge for the wrong reason.

## Artifacts and local preview

The AI quality artifact includes the generated `pr_quality_report.json`, companion Markdown, and
`pr_comment.md` below its CI report directory, alongside the source reports.
Artifact retention is controlled by the existing AI quality artifact configuration.

Build, preview, and validate locally without live providers:

```powershell
uv run python -m packages.evals.run_ci_report build --report-dir artifacts/ci/ai-quality --quality-policy-report artifacts/ci/ai-quality/phase3/quality_policy.json --output artifacts/ci/ai-quality/ci/pr_quality_report.json --format all --strict
uv run python -m packages.evals.run_ci_report render --input artifacts/ci/ai-quality/ci/pr_quality_report.json --format pr-comment --output artifacts/ci/ai-quality/ci/pr_comment.md
uv run python -m packages.evals.run_ci_report validate --input artifacts/ci/ai-quality/ci/pr_quality_report.json
```

## Troubleshooting

- Missing or malformed required JSON produces an infrastructure-error report and keeps the gate
  failed.
- Missing optional integrations are shown as skipped or unavailable; they are not represented as
  zero-valued metrics.
- If a comment is absent on a fork, use the job summary and artifact from the workflow run.
- If a report is long, use the artifact rather than expanding comment limits.
- Live providers, external judges, LangSmith, Phoenix, and OTLP remain disabled in CI.
